# fastapi-hexagonal-template

A production-grade FastAPI backend template built on **hexagonal architecture**
(ports & adapters), with the layer boundaries **enforced in CI** — not merely drawn in a
diagram. Designed to stay clean for humans and navigable for AI coding agents.

Python 3.13 · FastAPI · uv · async SQLAlchemy · Alembic · enforced with import-linter.

> **Status:** work in progress. This is an interim README; a fuller guide and docs site are
> planned.

## Why this template

Most "clean" or "hexagonal" FastAPI templates draw three layers and then let the domain
import the ORM anyway. Here the import direction is a build gate: `uv run poe arch` fails if
any inner layer reaches outward. The architecture cannot rot silently.

## Architecture

Hexagonal architecture, organized as three concentric layers. The rule: **imports point
inward only**. Outer layers know about inner layers, never the reverse — enforced by
[import-linter](https://import-linter.readthedocs.io/) (`uv run poe arch`).

```
┌─────────────────────────────────────────────────────────┐
│  infrastructure/        (outer — frameworks & drivers)   │
│  ┌───────────────────────────────────────────────────┐  │
│  │  application/        (middle — use cases)          │  │
│  │  ┌─────────────────────────────────────────────┐  │  │
│  │  │  domain/          (inner — pure business)    │  │  │
│  │  └─────────────────────────────────────────────┘  │  │
│  └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

```
src/app/
├── main.py                          # Composition root
├── domain/                          # Inner layer — pure business logic
│   ├── entities.py                  # Domain models (immutable Pydantic)
│   └── exceptions.py                # Domain-specific errors
├── application/                     # Middle layer — use cases
│   ├── use_cases/                   # Application services
│   └── ports/
│       ├── input/                   # Driving ports (what the app can do)
│       └── output/                  # Driven ports (what the app needs)
└── infrastructure/                  # Outer layer — frameworks & drivers
    ├── web/                         # FastAPI app, controllers, schemas, middleware
    ├── db/                          # SQLAlchemy engine, repositories, migrations
    ├── observability/               # Logging (loguru) + OpenTelemetry
    └── config.py                    # Settings (pydantic-settings)
```

- **domain/** — Pure business logic. No imports from `application/` or `infrastructure/`,
  no frameworks, no I/O. Entities are immutable; business-rule violations are domain
  exceptions, not HTTP status codes.
- **application/** — Use cases orchestrate domain objects to fulfil one operation each.
  `ports/input/` are the driving contracts controllers call; `ports/output/` are the driven
  contracts (e.g. repositories) that infrastructure implements. Depends on `domain/` only.
- **infrastructure/** — Every concrete technology choice: FastAPI controllers and DTO
  schemas, SQLAlchemy repositories, Alembic migrations, logging and tracing. Implements the
  ports defined in `application/`.

The import direction is checked in CI — a violation fails the build, so accidental coupling
can't slip in.

## Prerequisites

- [Python 3.13+](https://www.python.org/)
- [uv](https://docs.astral.sh/uv/)
- Docker (for integration tests and local Postgres)

## Installation

```bash
uv sync
uv run poe install-hooks
```

## Development

```bash
uv run poe dev      # Dev server (uvicorn --reload)
uv run poe check    # All checks: lint → typecheck → arch → test
uv run poe format   # Format code
```

## Database migrations

Migrations use [Alembic](https://alembic.sqlalchemy.org/) and live in
`src/app/infrastructure/db/migrations/versions/`.

```bash
uv run poe db-migrate                    # Apply all pending migrations
uv run poe db-revision -m "description"  # Autogenerate a new migration
uv run poe db-downgrade                  # Roll back one migration
```

Always review an autogenerated migration before applying it.

## Testing

```bash
uv run poe test         # pytest + coverage (enforces the coverage floor)
uv run poe test-quick   # pytest without coverage (fast iteration)
```

Unit tests need nothing external. Integration tests spin up Postgres via
[testcontainers](https://testcontainers.com/) (Docker required), or use `APP_DATABASE_URL`
if it is already set (e.g. in CI).

## Configuration

Settings are defined with pydantic-settings in `src/app/infrastructure/config.py`. All
environment variables use the `APP_` prefix (e.g. `APP_ENVIRONMENT=local`). See
`.env.example` for the full list.

## API documentation

When not running in production, interactive docs are served at `/docs` (Swagger UI) and
`/redoc`, with the OpenAPI schema at `/openapi.json`.
